# Phase 5 Sub-project: Unit Page + Lesson Viewer — Design

## Goal

Replace the `#/unit-1`..`#/unit-4` Coming Soon placeholders with a real Unit page (lesson list) and a new Lesson viewer (text + multiple-choice quiz) reachable at `#/unit-N/lesson-X`. Author real content for Unit 1's 3 lessons (titles already seeded in `json/course.json`). Completing a lesson's quiz updates `progress-store.js` (XP + that unit's progress percentage), which the already-shipped Dashboard/Roadmap/Progress/Achievements pages will automatically reflect since they all read the same store. Second and final currently-scoped Phase 5 sub-project (Units 2-4 content remains out of scope, per the original "one seed unit" decision).

## Architecture

**Router extension** (`js/router.js`): `handleRoute` currently resolves `routeId` (the full hash minus `#/`) via a single `routeTable[routeId]` lookup, redirecting to the default route if no match. A lesson hash like `#/unit-1/lesson-1` won't match any static table entry. Before that lookup, `handleRoute` gets a new check: a regex (`/^(unit-\d+)\/(.+)$/`) tested against `routeId`. On match, it dynamically imports the Lesson viewer directly (bypassing `routeTable`) and calls its `mount` with `{ unitId, lessonId }` as the `meta` argument — same three-argument `mount(mainEl, meta, isStale)` contract the Roadmap sub-project already established, so no new module contract is introduced. On no match, behavior is unchanged (existing `routeTable` lookup, unchanged for every other route). The `unit-1`..`unit-4` entries in `routeTable` are updated to load a new `js/modules/unit/unit.js` and each gets `meta: { unitId: "unit-1" }` (etc.) — reusing the router's existing meta pass-through mechanism (already used by `coming-soon.js` for `title`/`phase`) for a new purpose (telling the Unit page which unit it is), not adding a new mechanism.

**Sidebar highlighting**: `onRouteChange(routeId)` (passed into `initRouter` from `main.js`, driving `setActiveLink`) currently expects `routeId` to exactly match a sidebar link's `href`. For lesson routes, `handleRoute` calls `onRouteChange(unitId)` instead of the full `unit-N/lesson-X` string, so the sidebar keeps the parent unit highlighted while viewing any of its lessons — this is a one-line conditional in the same call site already there, not a new callback shape.

**Lesson content data**: new `json/lessons/<unit-id>-<lesson-id>.json` files (one per lesson, e.g. `json/lessons/unit-1-lesson-1.json`) — kept separate from `course.json` (which stays lightweight, titles-only, already used by both Roadmap and the new Unit page) so the Unit page's fetch doesn't have to download every lesson's full body just to list titles. Each lesson file has a `title` (redundant with `course.json`'s copy, used for a fallback/sanity-check display), a `sections` array of `{ heading, body }` pairs (body is plain text, rendered with `white-space: pre-wrap` the same way Notes/Bookmarks already handle multi-line user text — no markdown parser, keeping this dependency-free), and a `quiz` array of 2-3 `{ question, options: [...], correctIndex }` objects.

**New storage**: `js/storage/progress-store.js`'s `DEFAULT_STATE` gains `completedLessons: []` (an array of `"unit-1/lesson-1"`-style keys, mirroring the route format). A new exported `completeLesson({ lessonKey, unitId, unitProgressPercent, xpAward })`: if `lessonKey` is already in `completedLessons`, returns the current state unchanged (no double XP, no duplicate entry) — otherwise appends `lessonKey`, sets `unitProgress[unitId] = unitProgressPercent`, adds `xpAward` to `xp`, and persists via the same `setState`-style pattern every other store function uses. The **Lesson viewer** (not the store) computes `unitProgressPercent`, since it already has `course.json`'s lesson count for that unit in scope and the store stays a dumb persistence layer, consistent with every other store in this app (none of them contain business logic).

## Components

**Unit page** (`js/modules/unit/unit.js`): reads `meta.unitId`, fetches `course.json` (fetch-hardened identically to `roadmap.js`'s established pattern — try/catch, `res.ok`, shape check, graceful error render, `isStale` checks before each DOM-mutating path), finds the matching unit, renders `<h1>{unit.title}</h1>` + a `.unit__lessons` list. Each lesson row shows the title, a checkmark icon if `"${unitId}/${lesson.id}"` is in `getState().completedLessons`, and links to `#/${unitId}/${lesson.id}`. No locking (lessons within a unit are always clickable, consistent with the no-locking decision already made for units themselves).

**Lesson viewer** (`js/modules/lesson/lesson.js`): reads `meta.unitId`/`meta.lessonId`, fetches `json/lessons/${unitId}-${lessonId}.json` (same fetch-hardening pattern), renders the lesson `<h1>` + each section's heading/body, then a quiz form below: each question as a radio-button group, a "Submit" button. On submit: if every question's selected option matches `correctIndex`, calls `completeLesson(...)` and shows a success state (e.g. "Lesson complete! +XP", plus a link back to the Unit page); if any answer is wrong, shows inline per-question feedback (correct/incorrect) and lets the user change answers and resubmit — no attempt limit, no lockout, matching the earlier decision.

## Data Flow / State

Both new view modules are read-mostly: Unit page never writes; Lesson viewer writes exactly once, via `completeLesson(...)`, only on a fully-correct quiz submission, and only if that lesson wasn't already completed (idempotent — resubmitting a passed quiz doesn't re-award XP or duplicate the `completedLessons` entry, though the UI still shows the success state either way). Both follow the full-redraw-per-mount pattern already used everywhere else; the quiz form's per-question feedback is local render state inside `lesson.js`, not persisted.

## Error Handling

Both fetches (`course.json` in `unit.js`, the per-lesson JSON in `lesson.js`) use the identical hardened pattern from `roadmap.js`: try/catch, `res.ok` check, shape validation, `console.error` logging, and a graceful in-page error message instead of a blank page or unhandled rejection — plus `isStale()` checks before any DOM write, using the router's now-established `isStale` callback convention so rapid navigation away from a still-loading Unit/Lesson page can't corrupt `mainEl` (the exact bug class fixed during the Roadmap sub-project). If `unit.js` can't find a unit matching `meta.unitId` in the fetched `course.json` (shouldn't happen given the router only ever passes real unit ids, but defensively checked), it shows the same error state as a fetch failure rather than crashing on an `undefined` unit.

## Testing

No test framework (consistent with the rest of the project). Verification is manual: navigate `#/roadmap` → click Unit 1's "Start" link → confirm the Unit page lists all 3 lessons, none checked. Click a lesson, confirm the text renders, answer the quiz incorrectly, confirm inline wrong-answer feedback and no completion. Answer correctly, submit, confirm the success state, +XP, and that `localStorage.ailp:progress` now has the lesson key in `completedLessons` and the unit's `unitProgress` bumped to `1/3 ≈ 33%`. Navigate back to the Unit page, confirm that lesson now shows a checkmark. Complete all 3 lessons, confirm `unitProgress["unit-1"]` reaches 100 and the Dashboard/Roadmap/Progress pages (already shipped, unmodified) reflect it. Re-open an already-completed lesson and resubmit its (already-correct) quiz, confirm XP is not awarded a second time. Rapidly navigate away from a Unit or Lesson page mid-fetch (e.g. click a sidebar link immediately after clicking into a lesson), confirm no stale content flashes in and no console errors — exercising the `isStale` guard.

## Out of Scope (this sub-project)

- Real content for Units 2-4 (still "Coming Soon" placeholders in `course.json`, unchanged).
- Any content-authoring tooling/CMS — lesson JSON files are hand-authored, matching how `course.json` itself was authored.
- Lesson-to-lesson "Next"/"Previous" navigation within a unit (Unit page's lesson list already provides navigation; a linear "Next Lesson" button could be a small future addition but isn't required now).
- Retrying/resetting individual lesson completion (Settings' existing "Reset Progress" already resets everything, including `completedLessons`, via `resetProgress()`'s full-state overwrite — no new granular reset UI needed).
- Timed quizzes, partial credit, or multi-attempt scoring history — the quiz is binary (all-correct or not) with unlimited retries, per the earlier decision.
